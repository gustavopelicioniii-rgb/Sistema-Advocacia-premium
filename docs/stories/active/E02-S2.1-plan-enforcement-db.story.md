# Story E02-S2.1 — Plan Enforcement no Banco (Trigger + Tabela)

**Epic:** E-02 — Plan Enforcement Server-Side
**Status:** Ready
**Prioridade:** 🔴 CRÍTICO

## User Story

Como sistema, quero que o limite de processos por plano seja enforçado no banco de dados via trigger PostgreSQL, para que seja impossível bypassar o limite independente de como a inserção é feita.

## Contexto Técnico

Atualmente `useCreateProcesso` verifica o limite antes de inserir (frontend-only). Um usuário pode inserir diretamente via Supabase client, PostgREST API, ou Supabase Studio burlando o check.

Limite atual hardcoded em `useProfile.ts`:

```typescript
const PLAN_PROCESS_LIMITS = { start: 40, pro: 100, elite: 250 };
```

## SQL Reference

```sql
-- Tabela de planos
CREATE TABLE IF NOT EXISTS public.subscription_plans (
  plan_name TEXT PRIMARY KEY,
  process_limit INT NOT NULL,
  fee_limit INT
);
INSERT INTO public.subscription_plans VALUES
  ('start', 40, NULL),
  ('pro', 100, NULL),
  ('elite', 250, NULL)
ON CONFLICT DO NOTHING;

-- Função de check
CREATE OR REPLACE FUNCTION fn_check_process_limit()
RETURNS TRIGGER AS $$
DECLARE
  v_plan TEXT;
  v_limit INT;
  v_count INT;
BEGIN
  SELECT COALESCE(p.subscription_plan, 'start')
  INTO v_plan
  FROM public.profiles p
  WHERE p.id = NEW.owner_id;

  SELECT sp.process_limit INTO v_limit
  FROM public.subscription_plans sp
  WHERE sp.plan_name = v_plan;

  SELECT COUNT(*) INTO v_count
  FROM public.processos
  WHERE owner_id = NEW.owner_id;

  IF v_count >= v_limit THEN
    RAISE EXCEPTION 'Limite do plano % atingido (% processos). Atualize seu plano para cadastrar mais.', v_plan, v_limit;
  END IF;

  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

-- Trigger
CREATE TRIGGER trg_check_process_limit
  BEFORE INSERT ON public.processos
  FOR EACH ROW EXECUTE FUNCTION fn_check_process_limit();
```
